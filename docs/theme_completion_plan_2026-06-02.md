## 1. Краткий вывод

Приложение уже является рабочим расширенным MVP: есть авторизация, профиль, дневник питания, вода, вес, продукты, штрихкоды, рецепты, избранное, рекомендации, генерация плана питания, список покупок, аналитика, Health Connect и offline-очередь дневника.

При этом центральная цепочка темы пока реализована не полностью:

`выбранные продукты -> серверное хранение запасов -> подбор рецептов по наличию -> персональное ранжирование -> генерация плана с ограничениями -> сбор обратной связи -> обучение и оценка ML-модели`.

Сейчас:

- персональные рекомендации рецептов работают как rule-based scoring;
- генерация плана использует макроэвристику и случайность;
- экран холодильника хранит продукты только локально;
- подбор «из холодильника» сводится к поиску по одному часто используемому продукту;
- обучаемая ML-модель, датасет признаков и offline-оценка отсутствуют.

Для соответствия теме ВКР не нужно переписывать приложение. Нужно достроить единый серверный контур персонализации и показать измеримый переход от правил к ML-reranking.

## 4. Целевая пользовательская цепочка

Обязательный демонстрационный сценарий:

1. Пользователь указывает цель, вес, активность, диету, аллергии и исключения.
2. Добавляет продукты в «Мои продукты».
3. Приложение показывает рецепты, которые можно приготовить полностью или почти полностью.
4. Для каждого рецепта выводятся:
   - процент покрытия ингредиентов;
   - недостающие продукты;
   - персональный балл;
   - причина рекомендации;
   - КБЖУ и соответствие цели.
5. Пользователь генерирует план на несколько дней.
6. План учитывает профиль, ограничения, доступные продукты, КБЖУ и разнообразие.
7. Недостающие ингредиенты автоматически попадают в список покупок.
8. Пользователь добавляет выбранное блюдо в diary.
9. Событие принятия рекомендации сохраняется для последующего обучения модели.

Это и есть основной вертикальный срез ВКР.

## 5. Целевая архитектура

Backend остается modular monolith. Добавить отдельные логические модули:

- `Pantry`: продукты пользователя и остатки;
- `Recipe Candidate Generation`: рецепты по выбранным продуктам;
- `Recommendation`: rule-based baseline и ML-reranker;
- `Planning`: составление плана из ранжированных кандидатов;
- `Feedback`: события взаимодействия и обучающая выборка;
- `Data Quality`: валидация каталога и импортов.

Поток рекомендации:

`Profile + Pantry + Diary + Context -> Candidate Generation -> Hard Filters -> Feature Extraction -> Rule Score / ML Score -> Explanation -> Android`.

Hard filters всегда применяются до ML:

- аллергии;
- исключенные продукты;
- несовместимость с типом питания;
- некорректные или неполные рецепты;
- медицински небезопасные ограничения калорийности.

## 8. Алгоритм рекомендаций по продуктам

### Этап 1. Candidate generation

Для каждого рецепта вычислять:

- количество обязательных ингредиентов;
- количество найденных продуктов;
- покрытие ингредиентов;
- покрытие по массе;
- число недостающих ингредиентов;
- стоимость или сложность докупки;
- наличие скоро истекающих продуктов.

Базовая формула:

`coverage = matched_required_ingredients / required_ingredients`.

Отдельный бонус:

`expiry_bonus` для рецептов, использующих продукты с близким сроком годности.

### Этап 2. Hard filters

Исключить рецепты:

- с аллергенами;
- с исключенными продуктами;
- не соответствующие диете;
- без валидных ингредиентов и КБЖУ;
- с превышением допустимого числа недостающих продуктов.

### Этап 3. Rule-based baseline

Расширить существующий scoring:

- `pantry_coverage`;
- `missing_ingredients_penalty`;
- `nutrition_fit`;
- `goal_fit`;
- `meal_time_fit`;
- `prep_time_fit`;
- `favorite_fit`;
- `history_fit`;
- `novelty`;
- `repeat_penalty`;
- `expiry_bonus`.

Именно этот baseline нужен до ML: он дает fallback, объяснимость и точку сравнения.

## 9. ML-часть для ВКР

### 9.1 Постановка задачи

Тип задачи: learning-to-rank или бинарная классификация принятия рекомендации.

Объект: пара `(user, recipe, context)`.

Целевая переменная:

- `1`, если пользователь открыл и добавил рецепт в diary или план;
- `0`, если рецепт был показан, но проигнорирован или отклонен.

### 9.2 Признаки

Группы признаков:

- профиль: возрастная группа, цель, активность, диета;
- контекст: время суток, тип приема пищи, день недели;
- рацион: остаток калорий и КБЖУ за день;
- рецепт: калории, белок, жиры, углеводы, время готовки, кухня, рейтинг;
- pantry: процент покрытия, число недостающих продуктов, скоро истекающие продукты;
- история: просмотры, принятия, повторы, любимые кухни и продукты;
- разнообразие: сколько раз рецепт или кухня встречались за последние дни.

Не хранить чувствительные данные в feature snapshot без необходимости. Для ВКР достаточно обезличенного `user_id`.

### 9.3 Baseline и модель

Сравнить:

1. `Top Rated`: сортировка по рейтингу.
2. `Rule Score`: расширенный объяснимый скоринг.
3. `ML Reranker`: Logistic Regression, Random Forest или LightGBM/XGBoost.

Для первой версии практичнее Logistic Regression или Random Forest: проще встроить, объяснить и воспроизвести. Если датасет будет достаточно большим, использовать LightGBM/XGBoost как улучшение.

### 9.4 Интеграция

Рекомендуемый путь:

1. Обучение отдельным Python-скриптом из выгрузки PostgreSQL.
2. Сохранение модели и версии.
3. На первом этапе inference отдельным Python CLI/batch или небольшим FastAPI-сервисом.
4. Backend Spring вызывает ML-сервис только для reranking подготовленных кандидатов.
5. При недоступности ML используется `Rule Score`.

Для защиты допустим batch/offline inference, если продемонстрированы обучение, метрики и влияние модели на ранжирование.

### 9.5 Метрики

Offline:

- Precision@K;
- Recall@K;
- NDCG@K;
- ROC-AUC или F1 для классификационного baseline;
- доля рекомендаций, прошедших hard filters.

Online или имитационные:

- recommendation open rate;
- recommendation acceptance rate;
- plan meal completion rate;
- средний pantry coverage;
- среднее число недостающих ингредиентов.

### 9.6 Эксперимент

Использовать уже существующее разбиение:

- `CONTROL`: rule-based baseline;
- `EXPLORATION`: ML-reranking или скоринг с усиленным исследованием новых рецептов.

В отчете показать:

- состав датасета;
- матрицу признаков;
- способ разделения train/test по времени;
- сравнение метрик;
- несколько объяснимых примеров ранжирования.

### P3: не брать до закрытия P0

- Embeddings и нейросетевой ranking.
- Голосовой ввод.
- Сложная микросервисная декомпозиция.
- Большая рекомендательная платформа в реальном времени.

## 12. Что не следует называть ML в пояснительной записке

Чтобы формулировка была технически честной:

- текущий `PersonalizedRecommendationService` называть rule-based системой ранжирования;
- текущий `MealPlanService` называть эвристическим генератором плана;
- ML Kit image labeling на клиенте называть готовой моделью распознавания изображений, а не собственной моделью рекомендаций;
- собственной ML-частью считать обучаемый reranker, датасет и оценку метрик.
