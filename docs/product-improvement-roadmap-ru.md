# План улучшения продукта перед ВКР

Цель: довести приложение до состояния, где пользовательский путь понятен с регистрации, рекомендации объяснимы, база рецептов чистая, а расчеты КБЖУ и порций прозрачны.

## P0. Срочно исправить регистрацию

Проблема: при регистрации Android падает на ответе API:

```text
Expected start of the array '[', but had 'n' instead at path: $.data.favoriteCuisines
favoriteCuisines: null
```

Что сделать:

- На backend возвращать пустой массив вместо `null` для `favoriteCuisines`, `favoriteProductsIds`, `favoriteRecipesIds`, `excludedProductsIds` и похожих списков.
- На Android сделать DTO устойчивыми к `null`, если backend где-то все же вернет `null`.
- Проверить регистрацию через обычную форму и onboarding.
- Добавить тест/проверку API-ответа регистрации.

Критерий готовности:

- Новый пользователь регистрируется без ошибки.
- После регистрации пользователь попадает в приложение/onboarding.
- В логах нет `JsonDecodingException` по спискам предпочтений.

## P1. Понятный onboarding и выбор предпочтений

Проблема: сейчас не очевидно, где выбрать любимую или нелюбимую кухню, продукты и ограничения.

Что сделать:

- Добавить отдельный шаг onboarding "Предпочтения в еде".
- Разделить выборы:
  - любимые кухни;
  - кухни, которые не хочется видеть;
  - любимые продукты;
  - нежелательные продукты;
  - продукты, которые есть дома.
- Добавить короткий текст на экране: "Эти выборы влияют на план питания и рекомендации".
- В профиле сделать экран "Предпочтения", где это можно изменить позже.
- В генерации плана оставить быстрый необязательный выбор кухни как фильтр на конкретную генерацию.

Критерий готовности:

- Пользователь понимает, где настроить вкусы.
- В профиле видно, какие предпочтения уже выбраны.
- Генерация учитывает эти настройки.

## P2. Пересобрать аналитику вокруг главных пользовательских показателей

Проблема: вкладка аналитики есть, но пользователь не понимает, что главное и какие выводы делать.

Новая структура аналитики:

- Обзор:
  - средние калории за 7 дней;
  - белки/жиры/углеводы относительно цели;
  - вес и динамика;
  - активные дни ведения дневника;
  - выполнение плана питания.
- Питание:
  - калории по дням;
  - БЖУ;
  - недобор/перебор;
  - самые частые источники калорий.
- План питания:
  - сколько блюд пользователь оставляет;
  - сколько заменяет;
  - какие причины замен чаще всего;
  - какие кухни/ингредиенты повторяются.
- Персонализация:
  - какие действия пользователя влияют на ML;
  - "Вы чаще выбираете быстрые завтраки";
  - "Вы часто заменяете ужины на более легкие";
  - "Вы логировали блюда с высоким белком".

Критерий готовности:

- На первом экране аналитики видны 4-6 главных показателей.
- Пользователь понимает, почему рекомендации меняются.
- Можно объяснить на ВКР, какие данные собираются для ML.

## P3. Прозрачность ML и действий пользователя

Проблема: пользователю и комиссии не видно, на основе чего работает ML-подбор.

Что сделать:

- В аналитике добавить блок "Что приложение узнало о вас".
- Показывать простыми фразами:
  - "Вы закрепляли быстрые завтраки";
  - "Вы удаляли блюда с долгим временем готовки";
  - "Вы чаще логировали блюда с белком выше 25 г";
  - "Вы выбирали итальянскую кухню чаще остальных".
- Для каждого блюда в плане показывать причину выбора:
  - цель по калориям;
  - белок;
  - продукты дома;
  - любимая кухня;
  - разнообразие недели;
  - прошлые действия пользователя.

Критерий готовности:

- Пользователь видит не "магическую рекомендацию", а объяснение.
- На защите можно показать связь: действие пользователя -> обучающий сигнал -> изменение рекомендаций.

## P4. Ревизия базы рецептов

Проблема: в базе есть рецепты, которые не являются полноценными блюдами: "Крем для торта без миксера", "Тесто для вареников без яиц". Есть рецепты с фразами вроде "подавайте с гарниром", но гарнир не входит в расчет КБЖУ.

Что сделать:

- Добавить классификацию рецептов:
  - полноценное блюдо;
  - гарнир;
  - соус/крем;
  - тесто/заготовка;
  - десерт;
  - напиток;
  - компонент блюда.
- Добавить поле `recipe_role` или аналог:
  - `FULL_MEAL`;
  - `SIDE_DISH`;
  - `COMPONENT`;
  - `SAUCE`;
  - `DESSERT`;
  - `SNACK`.
- В генерации плана использовать только `FULL_MEAL`, `DESSERT`, `SNACK` в подходящих слотах.
- Компоненты не предлагать как самостоятельный обед/ужин.
- Сделать админский список подозрительных рецептов:
  - название содержит "крем", "тесто", "соус", "маринад";
  - калорийность слишком низкая/высокая;
  - нет белков/жиров/углеводов;
  - нет ингредиентов;
  - текст содержит "подавайте с гарниром".

Критерий готовности:

- Генератор не предлагает заготовки как полноценный прием пищи.
- У рецепта понятно, является ли он самостоятельным блюдом.
- Рецепты с гарниром помечены как требующие дополнения.

## P5. Гарниры и составные блюда

Проблема: если рецепт говорит "подавайте с гарниром", расчет КБЖУ неполный.

Что сделать:

- Добавить признак `requires_side_dish`.
- Добавить рекомендуемые гарниры для блюда.
- Сделать составное блюдо:
  - основное блюдо;
  - гарнир;
  - соус по желанию.
- При генерации плана учитывать КБЖУ всего набора.
- В карточке показывать: "КБЖУ рассчитано без гарнира" или "КБЖУ рассчитано вместе с гарниром".

Критерий готовности:

- Пользователь понимает, входит ли гарнир в расчет.
- Генератор не занижает калории блюда из-за отсутствующего гарнира.

## P6. Прозрачный калькулятор КБЖУ

Проблема: пользователь не может проверить, как рассчитана пищевая ценность блюда.

Что сделать:

- В карточке рецепта добавить экран "Расчет КБЖУ".
- Показывать таблицу ингредиентов:
  - продукт;
  - количество в граммах;
  - калории на 100 г;
  - белки/жиры/углеводы на 100 г;
  - вклад продукта в блюдо.
- Показывать формулы:
  - калории ингредиента = граммы / 100 * калории на 100 г;
  - белки = граммы / 100 * белки на 100 г;
  - КБЖУ блюда = сумма по ингредиентам;
  - КБЖУ порции = КБЖУ блюда / количество порций.
- Показывать предупреждения:
  - нет веса ингредиента;
  - нет КБЖУ продукта;
  - используется примерная оценка;
  - гарнир не включен.

Критерий готовности:

- Пользователь может понять, почему у блюда такие калории.
- Ошибки данных становятся видимыми, а не скрытыми.
- На ВКР можно показать формулы и прозрачность расчетов.

## P7. Порции, семья и meal prep

Проблема: сейчас не очевидно, сколько порций в рецепте, сколько пользователь съел и как готовить на несколько человек или на несколько дней.

Что сделать:

- В каждом рецепте явно показывать:
  - количество порций;
  - вес одной порции;
  - КБЖУ на 1 порцию;
  - КБЖУ на 100 г.
- При добавлении в дневник дать выбор:
  - 0.5 порции;
  - 1 порция;
  - 2 порции;
  - граммы вручную.
- В плане питания добавить сценарии:
  - готовлю только для себя;
  - готовлю на семью;
  - готовлю на несколько дней.
- Для семьи:
  - указать количество человек;
  - пересчитать список покупок;
  - оставить дневник пользователя только по его съеденной порции.
- Для meal prep:
  - приготовить 3-4 порции;
  - распределить остатки на другие дни;
  - не считать это повтором как ошибку, если пользователь сам выбрал meal prep.

Критерий готовности:

- Пользователь понимает, сколько готовить и сколько съедать.
- Список покупок масштабируется под семью.
- План может осознанно повторять блюдо, если это meal prep.

## P8. Изменения в БД

Предварительно нужны новые/уточненные сущности и поля:

- `recipes`:
  - `recipe_role`;
  - `requires_side_dish`;
  - `servings_count`;
  - `portion_weight_g`;
  - `nutrition_calculation_status`;
  - `nutrition_notes`.
- `recipe_ingredients`:
  - точное количество;
  - единица измерения;
  - вес в граммах;
  - обязательность ингредиента;
  - признак "по вкусу".
- `recipe_side_dishes`:
  - основное блюдо;
  - рекомендуемый гарнир;
  - коэффициент порции.
- `meal_plan_settings` или профиль пользователя:
  - готовлю для себя/семьи;
  - количество человек;
  - meal prep включен;
  - допустимое число повторов блюда.
- `nutrition_calculation_audit`:
  - рецепт;
  - версия расчета;
  - список проблем;
  - дата пересчета.

Критерий готовности:

- Схема БД поддерживает прозрачный расчет, гарниры, порции и семейный сценарий.

## P9. Очистка и пересчет данных

Что сделать:

- Написать скрипт аудита рецептов.
- Найти мусорные/компонентные рецепты.
- Назначить `recipe_role`.
- Пересчитать КБЖУ по ингредиентам.
- Пометить рецепты с неполными данными.
- Скрыть неподходящие рецепты из генерации плана.

Критерий готовности:

- В генерации остаются только валидные блюда.
- У спорных рецептов есть предупреждение.

## Рекомендуемый порядок работ

1. Исправить регистрацию.
2. Сделать понятный выбор предпочтений в onboarding/profile.
3. Переработать аналитику: обзор + персонализация + ML-сигналы.
4. Добавить `recipe_role` и скрыть мусорные рецепты из генерации.
5. Сделать экран прозрачного расчета КБЖУ.
6. Доработать порции и добавление нескольких порций в дневник.
7. Добавить family/meal-prep настройки.
8. Добавить гарниры и составные блюда.
9. Провести полный аудит БД рецептов.

- Регистрация и onboarding.
- Пользователь выбирает цель, предпочтения и продукты дома.
- Приложение генерирует план питания.
- У блюда видно объяснение рекомендации.
- У блюда видно прозрачный расчет КБЖУ.
- Пользователь меняет/логирует блюдо.
- Аналитика показывает, какие действия влияют на ML.
- Приложение постепенно адаптирует план питания под пользователя.
